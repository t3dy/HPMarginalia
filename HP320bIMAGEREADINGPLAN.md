## The Problem

We have 674 high-quality manuscript photographs across two copies of the
Hypnerotomachia Poliphili. Our entire pipeline currently treats these images
as opaque blobs — matched to database records by filename arithmetic, never
actually read. This means:

1. We don't know what's actually on most pages (only 27 of 196 BL photos
   have been visually inspected)
2. We can't transcribe the handwritten marginalia visible in the photographs
3. We can't systematically identify alchemical symbols, woodcuts, or
   annotation density beyond what Russell chose to document in his thesis
4. We can't verify our concordance matches against what the images show

Image reading is not an enrichment feature. It is the missing verification
layer in the concordance pipeline. Without it, every match between a
dissertation reference and a manuscript photograph rests on arithmetic
assumptions that have never been checked against what the photographs
actually show.

## Image Source Boundary

### The Two Image Corpora

This project maintains two distinct image sets for each manuscript. They
serve different purposes and must never be confused.

**Master / Original Image Corpus**

The research-quality photographs from which all analysis, verification,
and reading work must be performed.

| Collection | Path | Files | Avg Size | Quality |
|-----------|------|-------|----------|---------|
| BL C.60.o.12 | `3 BL C.60.o.12 Photos.../3 BL C.60.o.12 Photos/` | 196 | ~4 MB | HIGH — full-resolution JPEG scans |
| Siena O.III.38 | `Siena O.III.38 Digital Facsimile.../Siena O.III.38 Digital Facsimile/` | 478 | ~500 KB | MEDIUM — digital facsimile |

These paths are stored in `manuscripts.image_dir` and resolved via
`images.master_path` (added 2026-03-20).

**Web / Display Derivatives**

Compressed copies generated by `compress_images.py` for GitHub Pages
deployment. These exist solely for browser display.

| Collection | Path | Files | Avg Size | Quality |
|-----------|------|-------|----------|---------|
| BL compressed | `site/images/bl/` | 196 | ~200 KB | LOW — 800px wide, 75% JPEG |
| Siena copy | `site/images/siena/` | 478 | ~500 KB | MEDIUM — copied as-is |

These paths are stored in `images.web_path` and used by `build_site.py`
for `<img>` tags.

### Enforcement Rules

The distinction between master and web images is not advisory. It is a hard
boundary that must be enforced in code:

1. **Any script that reads image bytes for analysis, verification, OCR,
   vision model input, or comparison MUST resolve the path from
   `images.master_path`.** No exceptions.

2. **Any script that generates HTML or JSON for web display MUST resolve
   the path from `images.web_path` (or `COALESCE(i.web_path, i.relative_path)`
   for backward compatibility).** No exceptions.

3. **Scripts that accept an image path argument MUST validate that the path
   does not begin with `site/images/` or `images/bl/` or `images/siena/`
   before performing any analytical operation.** If the path resolves to a
   web derivative, the script should fail with an explicit error:

   ```
   ERROR: Refusing to read web-derivative image for analysis.
   Path 'site/images/bl/C_60_o_12-014.jpg' is a compressed display copy.
   Use master_path '3 BL C.60.o.12 Photos.../C_60_o_12-014.jpg' instead.
   ```

4. **The `read_images.py` script (to be built) MUST assert on startup that
   the master image directories exist and are non-empty.** If either
   directory is missing, the script must refuse to run, not fall back to
   web copies.

### Why This Matters

The BL master images are ~4 MB each. The web derivatives are ~200 KB — a
95% reduction in image data. At web resolution, handwriting is illegible,
alchemical symbols are a few indistinct pixels, and fine print cannot be
read. Sending a web derivative to a vision model is worse than useless:
it would produce confident but wrong transcriptions based on insufficient
visual data.

The Siena images happen to be identical between master and web (the
originals were already small enough to copy as-is). But the boundary must
still be enforced for both collections, because (a) the Siena originals
could be upgraded later and (b) the principle is that analysis reads from
source, display reads from derivatives.

## The Image Reading Layer

### The Problem with Direct Insertion

The existing plan routed vision outputs directly into canonical scholarly
tables (`annotations`, `woodcuts`, `symbol_occurrences`). This is
architecturally wrong for three reasons:

1. **Auditability.** If a vision model misreads a page number or invents
   a transcription, there is no way to review what it originally said
   versus what entered the database. The raw output is lost.

2. **Separation of evidence.** A vision model's reading of a photograph is
   a different kind of evidence than Russell's scholarly description of the
   same folio. Mixing them in the same table without a reviewable
   intermediate layer makes it impossible to distinguish machine-read data
   from corpus-extracted data at audit time.

3. **Non-destructive review.** If all 196 BL images are read in one batch,
   someone needs to be able to review the results before any of them touch
   canonical tables. Direct insertion makes this all-or-nothing.

### Design Decision: Staging Layer vs Dedicated Table

There are two reasonable approaches. Both preserve auditability. The choice
depends on whether the project wants to keep vision outputs as a permanent
queryable layer or treat them as transient staging data.

**Option A: Dedicated `image_readings` table (PREFERRED)**

A new table that holds all structured outputs from vision model reading.
Each row represents one reading of one image, with the raw JSON preserved.

```sql
CREATE TABLE image_readings (
    id INTEGER PRIMARY KEY,
    image_id INTEGER REFERENCES images(id),
    phase INTEGER NOT NULL,               -- 1, 2, 3, or 4
    model TEXT NOT NULL,                   -- 'sonnet-4.6', 'opus-4.6', etc.
    raw_json TEXT NOT NULL,                -- Full API response, preserved verbatim
    page_number_read INTEGER,             -- Phase 1: what the model read
    page_number_expected INTEGER,         -- From offset formula or filename
    page_number_match BOOLEAN,            -- Deterministic comparison
    has_woodcut BOOLEAN,                  -- Phase 1
    woodcut_description TEXT,             -- Phase 1
    has_annotations BOOLEAN,              -- Phase 2
    annotation_density TEXT,              -- Phase 2: LIGHT/MODERATE/HEAVY
    annotation_locations TEXT,            -- Phase 2: JSON array
    languages_detected TEXT,              -- Phase 2: JSON array
    legible_fragments TEXT,               -- Phase 2: JSON array
    deep_reading_json TEXT,               -- Phase 3: full structured reading
    concordance_status TEXT CHECK(concordance_status IN (
        'CONFIRMED', 'DISCREPANCY', 'UNVERIFIED'
    )) DEFAULT 'UNVERIFIED',
    reviewed BOOLEAN DEFAULT 0,
    reviewed_by TEXT,
    reviewed_at TEXT,
    promoted_to TEXT,                     -- e.g. 'annotations:42, woodcuts:19' after review
    notes TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);
```

**Benefit:** Vision outputs are permanently queryable. You can ask "what did
the model see on photo 041?" at any time. Review status is tracked per
reading. Promotion to canonical tables is recorded. The `raw_json` column
preserves the exact API response for re-parsing if the schema evolves.

**Risk:** Adds a 23rd table to a schema that already has 22. Adds a pipeline
step (review + promote) before data reaches canonical tables.

**Option B: File-based staging with ingest script (CONSERVATIVE)**

Vision outputs go to `staging/image_readings/` as JSON files (already in
the current plan). A separate `ingest_image_readings.py` script reads
these files, validates them, and inserts qualifying records into canonical
tables with `source_method='VISION_MODEL'`.

**Benefit:** No new tables. Follows the existing staging pattern used by
reading packets. Simpler schema.

**Risk:** Staging files are not queryable by SQL. Harder to correlate
image readings with matches and annotations. Review state lives in the
filesystem, not the database. If staging files are deleted, the audit
trail is lost.

### Recommendation

**Option A is preferred.** The image reading pipeline will produce structured
evidence that needs to be compared against existing concordance data,
reviewed for accuracy, and selectively promoted to canonical tables. This
is a relational problem. A table handles it better than a directory of
JSON files.

The `image_readings` table is not speculative. It holds concrete outputs
from a concrete pipeline. It follows the project's provenance model
(source_method, confidence, review_status). And it solves the auditability
problem that direct insertion creates.

**If the project decides against a new table**, Option B works. But the
ingest script must then maintain its own review-state tracking (e.g., a
`staging/image_readings/review_log.json`) and must never delete staging
files after ingestion.

### Data Flow with Image Reading Layer

```
Master images (via images.master_path)
    |
    v
read_images.py (sends to Claude Vision API)
    |
    v
Raw JSON responses → staging/image_readings/ (backup)
    |
    v
image_readings table (structured, reviewable, queryable)
    |                                          |
    v                                          v
Concordance verification          Human review (reviewed=1)
(compare against matches,              |
 flag discrepancies)                   v
                                  Promote to canonical tables
                                  (annotations, woodcuts,
                                   symbol_occurrences)
                                  with source_method=VISION_MODEL,
                                  confidence=PROVISIONAL,
                                  needs_review=1
```

The key principle: **vision outputs exist as a distinct layer before they
become scholarly data.** They can be queried, compared, reviewed, and
promoted — or rejected — without ever touching canonical tables until a
human (or a deterministic validation rule) says they should.

## Image Reading as Concordance Verification

### The Feedback Loop

Image reading is not a separate enrichment feature bolted onto the
concordance. It is the verification step the concordance has always lacked.
The concordance currently works like this:

```
Russell's thesis says: "Hand B annotates b6v with alchemical symbols"
    → extract signature ref: b6v
    → look up b6v in signature_map: folio 28
    → BL offset: photo 28 + 13 = photo 041
    → match to images.filename = C_60_o_12-041.jpg
    → confidence: MEDIUM (never verified against the actual image)
```

With image reading, the loop closes:

```
1. EXPECTATION (from database):
   - annotations table says: b6v has alchemical annotations (Hand B)
   - matches table says: b6v → photo 041, confidence MEDIUM
   - signature_map says: b6v = folio 28

2. OBSERVATION (from vision model reading master image):
   - Photo 041 shows: page number 28 (CONFIRMED)
   - Photo 041 shows: woodcut present — elephant and obelisk (NEW DATA)
   - Photo 041 shows: annotations HEAVY, right margin + above woodcut
   - Photo 041 shows: legible fragment "bellua" (MEDIUM confidence)

3. COMPARISON (deterministic):
   - Expected page 28, read page 28 → MATCH
   - Annotations expected, annotations found → CONFIRMED
   - Woodcut not previously recorded → NEW DISCOVERY
   - Text "bellua" not in annotations table → POTENTIAL NEW ANNOTATION

4. OUTCOME:
   - match confidence: MEDIUM → upgrade candidate (pending review)
   - woodcut: insert into image_readings, flag for promotion to woodcuts table
   - text fragment: insert into image_readings, flag for review
   - overall concordance_status: CONFIRMED
```

### What Gets Logged, Not Auto-Corrected

The following discrepancies must be logged in `image_readings` (or
`staging/image_readings/`) and flagged for human review. They must NEVER
trigger automatic database updates:

**Page number mismatch:**
```
DISCREPANCY: Photo 041 — expected page 28 (from offset formula),
vision model reads page 29. Possible causes: offset error at this
point, misread by model, inserted/cancelled leaf in 1545 edition.
Action: LOG, do not update images.folio_number.
```

**Expected annotation absent:**
```
DISCREPANCY: Photo 041 — annotations table says b6v has Hand B
marginalia, but vision model detects NO annotations on this page.
Possible causes: wrong photo-folio mapping, annotation on adjacent
page, model failed to detect faint marks.
Action: LOG, do not delete annotation record.
```

**Unexpected heavy marginalia:**
```
DISCREPANCY: Photo 077 — no annotations table record for this folio,
but vision model detects HEAVY marginalia in right margin.
Possible causes: Russell did not document this page, annotation is
by a non-alchemical hand, or this represents a new finding.
Action: LOG as potential new annotation site, flag for review.
```

**Woodcut present where none recorded:**
```
DISCREPANCY: Photo 100 — no woodcut record for this folio, but
vision model detects a woodcut illustration (grotesque frieze).
Action: LOG in image_readings, flag for promotion to woodcuts table
after review.
```

**Woodcut expected but absent:**
```
DISCREPANCY: Photo 055 — woodcuts table says c5v has a woodcut,
but vision model sees text only (no illustration on this page).
Possible causes: wrong folio mapping, woodcut on adjacent page,
model error.
Action: LOG, do not delete woodcut record.
```

### Concordance Confidence Updates

Image reading can support confidence upgrades, but the upgrade path must
be explicit:

| Current Confidence | Image Reading Result | New Confidence | Requires |
|-------------------|---------------------|---------------|----------|
| MEDIUM | Page number confirmed + annotations match | MEDIUM (upgrade candidate) | Human review to reach HIGH |
| MEDIUM | Page number mismatch | MEDIUM + DISCREPANCY flag | Human investigation |
| PROVISIONAL | Any confirmed reading | PROVISIONAL (upgrade candidate) | Human review |
| HIGH | N/A | HIGH (no downgrade from vision) | — |

Rule: **Vision reading can flag upgrade candidates. It cannot execute
upgrades.** Confidence changes require human review. Vision reading can
never downgrade an existing HIGH confidence match.

## Provenance Discipline

### Rules (non-negotiable)

1. **No vision output is ever auto-inserted as VERIFIED.** All image-reading
   results enter the system as `PROVISIONAL` with `needs_review=1`.

2. **Canonical scholarly tables are never silently overwritten.** If a vision
   model reads something that contradicts an existing `annotations` or
   `woodcuts` record, the discrepancy is logged. The existing record stands
   until a human resolves the conflict.

3. **Discrepancies are preserved for audit.** A discrepancy between what the
   vision model reads and what the database says is itself valuable data.
   It must be stored, not discarded. The `image_readings` table (or staging
   file) is the permanent record of what the model actually reported.

4. **Source method is always recorded.** Every datum that enters the database
   from vision reading carries `source_method='VISION_MODEL'`. This is
   distinct from `CORPUS_EXTRACTION` (Russell's thesis), `LLM_ASSISTED`
   (enrichment from text), and `HUMAN_VERIFIED` (expert review).

5. **Raw API responses are preserved.** The exact JSON returned by the vision
   API is stored in `image_readings.raw_json` (or as a file in
   `staging/image_readings/`). If the extraction schema changes, the raw
   response can be re-parsed without re-running the (costly) API call.

### How Each Rule Is Enforced (prose vs infrastructure)

These rules are correct, but as of v2 they exist only as English sentences
in this document. A script that doesn't read the plan can violate every
one. The following table shows what infrastructure is needed to make each
rule structural rather than advisory.

| Rule | Current Enforcement | Required Infrastructure |
|------|-------------------|----------------------|
| 1. No auto-VERIFIED | PROSE ONLY | `image_readings` table: DEFAULT 'PROVISIONAL', `needs_review` DEFAULT 1. Promotion script refuses unless `reviewed=1`. |
| 2. No silent overwrite | PROSE ONLY | Promotion script checks for existing record with non-VISION_MODEL source. Refuses if found. |
| 3. Discrepancies preserved | PROSE ONLY | `image_readings.concordance_status` column with CHECK constraint. Comparison script populates it. No DELETE without explicit flag. |
| 4. source_method recorded | **BLOCKED** — `annotations.source_method` CHECK rejects 'VISION_MODEL' | Schema migration: recreate `annotations` with 'VISION_MODEL' added to CHECK. Same for any target table. |
| 5. Raw JSON preserved | PROSE ONLY (table doesn't exist) | `image_readings.raw_json TEXT NOT NULL` — schema enforces non-null. Backup to `staging/` as belt-and-suspenders. |

### Schema Gaps That Must Be Fixed Before Any Phase Runs

These are not future wishes. They are bugs in the current schema that will
cause INSERT failures if the plan is executed as written.

**1. `annotations.source_method` rejects 'VISION_MODEL'**

Current CHECK constraint:
```sql
source_method TEXT CHECK(source_method IN (
    'MANUAL_TRANSCRIPTION','PDF_EXTRACTION','LLM_ASSISTED','INFERRED'
)) DEFAULT 'INFERRED'
```

Fix: Recreate the table with `'VISION_MODEL'` added. SQLite does not
support ALTER TABLE to modify CHECK constraints. Migration script must:
create temp table, copy data, drop original, create new with expanded
CHECK, copy back, update schema_version.

**2. `matches.confidence` rejects 'PROVISIONAL'**

Current CHECK constraint:
```sql
confidence TEXT CHECK(confidence IN ('HIGH','MEDIUM','LOW'))
```

The plan's provenance model uses PROVISIONAL as the default for vision
outputs. If a match is flagged for confidence review, the comparison
script cannot store the intermediate state. Fix: add 'PROVISIONAL' to
the CHECK constraint via the same migration pattern.

**3. `woodcuts.source_method` and `symbol_occurrences.source_method` have
no CHECK constraint**

These columns accept any string, which means 'VISION_MODEL' will INSERT
successfully — but also means typos like 'VISION_MODLE' will too. Add
CHECK constraints matching the expanded annotations vocabulary.

**4. `image_readings` table does not exist**

The plan's entire data flow routes through this table, but it has never
been created. This is prerequisite #1.

### Shared Infrastructure That Must Exist

**`scripts/image_utils.py`** — A shared Python module, not per-script
reimplementation. Contains:

```python
def resolve_master_path(conn, image_id) -> Path:
    """Return the master_path for an image. Raises if missing."""

def assert_not_web_derivative(path: Path) -> None:
    """Raise ValueError if path is under site/images/."""

def assert_master_dirs_exist(conn) -> None:
    """Check that all manuscripts.image_dir directories exist on disk.
    Raise SystemExit if any are missing."""
```

Every image-touching script imports from `image_utils`. Path validation
is not a suggestion that lives in a planning document — it is a function
call that runs before any image byte is read.

### Validation Layers

| Boundary | What Crosses It | Validation Required |
|----------|----------------|-------------------|
| Vision API → `image_readings` table | Raw structured output | Schema validation only. All fields nullable except image_id and raw_json. |
| `image_readings` → `annotations` | Transcribed text, annotation_type | Human review. `source_method='VISION_MODEL'`, `confidence='PROVISIONAL'`, `needs_review=1`. Promotion script refuses if existing non-vision record exists. |
| `image_readings` → `woodcuts` | Woodcut descriptions | Human review. Cross-check against known 172-woodcut inventory. |
| `image_readings` → `symbol_occurrences` | Detected alchemical symbols | Human review. PROVISIONAL only. Expert verification required. |
| `image_readings` → `images.folio_number` | Page number verification | Discrepancy log only. Do not auto-correct. |
| Batch cost → budget | API token spend | Hard cap per run. Alert at 80% budget. |

## Proposed Pipeline

### Phase 0: Infrastructure (prerequisite — no API calls)

**Purpose:** Build the schema, shared utilities, and migration scripts
that all reading phases depend on. This phase costs $0 and takes ~1 hour.

**Scope:** Database schema changes, shared Python module, staging directories.
No vision API calls. No image reading.

**Deliverables:**

| # | Deliverable | Acceptance Test |
|---|------------|-----------------|
| 1 | `image_readings` table created | `sqlite3 db/hp.db ".schema image_readings"` succeeds |
| 2 | `annotations.source_method` CHECK expanded to include `'VISION_MODEL'` | `INSERT INTO annotations (..., source_method) VALUES (..., 'VISION_MODEL')` succeeds |
| 3 | `matches.confidence` CHECK expanded to include `'PROVISIONAL'` | INSERT succeeds |
| 4 | `woodcuts.source_method` and `symbol_occurrences.source_method` get CHECK constraints | Typo inserts rejected |
| 5 | `scripts/image_utils.py` with `resolve_master_path()`, `assert_not_web_derivative()`, `assert_master_dirs_exist()` | `python -c "from scripts.image_utils import resolve_master_path"` succeeds |
| 6 | `staging/image_readings/bl/phase1/` through `siena/phase4/` directories | Directories exist |
| 7 | 27 previous readings backfilled into `image_readings` (phase=0, from IMAGEIDENTIFICATION.md) | `SELECT COUNT(*) FROM image_readings WHERE phase=0` = 27 |
| 8 | `schema_version` updated | Migration recorded |

**Why this must come first:** Every provenance rule in this plan, every
data flow arrow, every safety check depends on infrastructure that does
not currently exist. Phase 1 without Phase 0 would hit CHECK constraint
failures, write to nonexistent tables, and rely on per-script validation
that nobody has written. Phase 0 is the difference between a plan that
works and a plan that fails on first INSERT.

### Phase 1: Visual Ground Truth (BL only)

**Purpose:** Establish machine-readable visual evidence for every BL
photograph, creating the foundation for concordance verification.

**Scope:** All 196 BL master images (via `images.master_path`)
**Cost estimate:** ~$2-3 (Sonnet vision, ~196 calls, ~500 tokens output each)
**Time:** ~1 hour

**For each image, extract:**
```json
{
  "photo_number": 14,
  "page_number_visible": true,
  "page_number_read": 1,
  "page_number_expected": 1,
  "page_number_match": true,
  "has_woodcut": false,
  "woodcut_description": null,
  "page_type_observed": "TEXT"
}
```

**Deliverables:**
- Ground-truth folio mapping: vision-read page numbers compared against
  offset formula for every BL photo
- Complete woodcut inventory for the BL copy (estimated ~45 woodcuts)
- Discrepancy log for any page where read != expected
- Identification of non-text pages (covers, flyleaves, blanks)

**Why first:** This phase produces the highest-value data at the lowest cost.
It converts the offset assumption (`photo - 13 = folio`) from a formula
verified at 27 points into a formula verified at 196 points — or reveals
where it breaks. Every subsequent phase depends on knowing which photo
shows which page.

**DB writes:** Insert into `image_readings` (phase=1). No direct writes to
canonical tables. Discrepancies flagged for review.

### Phase 2: Coverage Mapping (BL only)

**Purpose:** Map the full annotation landscape of the BL copy — which pages
have marginalia, how dense, what languages — independent of what Russell
chose to document.

**Scope:** All 196 BL master images
**Cost estimate:** ~$3 (can combine with Phase 1 to save cost)
**Time:** ~1 hour (or free if combined with Phase 1)

**For each image, extract:**
```json
{
  "has_annotations": true,
  "annotation_density": "HEAVY",
  "annotation_locations": ["right_margin", "bottom"],
  "languages_detected": ["Latin"],
  "legible_text_fragments": ["bellua", "Cortes"]
}
```

**Deliverables:**
- Full annotation density map across the entire BL copy
- List of pages with annotations Russell did NOT document
  (cross-referenced against `annotations` table)
- Language distribution across the marginalia
- Pages ranked by annotation density for Phase 3 targeting

**Why second:** Phase 2 answers "where should we look closely?" Its outputs
directly determine which pages are worth the higher cost of Phase 3 deep
reading. Without Phase 2, Phase 3 would either read every page (wasteful)
or rely only on Russell's selection (incomplete).

**DB writes:** Insert into `image_readings` (phase=2). No direct writes to
canonical tables.

### Phase 3: Targeted Deep Reading (BL, ~20 key pages)

**Purpose:** Produce detailed structured readings of the most important

## Technical Architecture

### Script: `scripts/read_images.py` (to be built)

```
read_images.py
  --phase [1|2|3|4]       Which phase to run
  --source bl|siena        Which manuscript
  --start N                Resume from photo N
  --dry-run                Show what would be processed, don't call API
  --budget DOLLARS          Hard cap on API spend
  --model sonnet|opus      Which vision model (Opus for Phase 3, Sonnet for 1-2)
```

**Startup validation (must pass before any API call):**
1. Assert master image directory exists and is non-empty
2. Assert no image path resolves to `site/images/`
3. Assert API key is configured and has sufficient budget
4. Assert `image_readings` table exists (or staging directory, if Option B)
5. Report: N images to process, estimated cost, estimated time

**Per-image flow:**
1. Resolve `master_path` from `images` table
2. Validate path is not a web derivative
3. Load image bytes from master path
4. Send to vision API with phase-appropriate prompt
5. Write raw JSON to `staging/image_readings/{ms}/{phase}/{photo}.json`
6. Parse response into structured fields
7. Insert into `image_readings` table with `concordance_status='UNVERIFIED'`
8. Run deterministic comparison against expected values (offset, annotation
   records, match confidence)
9. Set `concordance_status` to `CONFIRMED` or `DISCREPANCY`
10. Log discrepancies to stdout and to `notes` field

regardless of whether Option A or Option B is chosen for the database layer:

## Role of Multimodal RAG

### What It Would Enable

A multimodal retrieval-augmented generation layer would allow querying
across all evidence types simultaneously. For a given folio, a RAG query
could retrieve together:

- The master photograph (image embedding)
- Russell's dissertation excerpt discussing that folio (text chunk)
- The existing `annotations` record (structured data)
- The `annotator_hands` profile for the attributed hand (structured data)
- The `image_readings` record from the vision pipeline (structured output)
- The `folio_descriptions` analysis if one exists (prose)

This would enable questions like: "Show me every folio where the vision
model detected heavy annotations but Russell did not document any" or
"Find pages where the BL alchemist's annotations co-occur with specific
woodcut subjects."

### What It Is Not

Multimodal RAG is **not a substitute for source image quality.** Embedding
a compressed 200KB web image produces a worse vector than embedding the
4MB master. The image source boundary applies to embedding pipelines just
as it applies to vision model reading.

Multimodal RAG is **not the first step.** It is most useful after the image
reading pipeline has produced structured evidence units (the `image_readings`
records). Without those records, there is nothing structured to retrieve
alongside the images and text.

Multimodal RAG is **not required for initial verification.** Phases 1-3 of
the image reading pipeline operate without any retrieval infrastructure.
They read images, compare outputs against database records, and log results.
RAG becomes valuable when you want to ask cross-cutting questions across
the accumulated evidence.

### When to Build It

After Phases 1-3 are complete and reviewed, the project will have:
- 196 structured image readings (BL)
- A complete woodcut inventory
- An annotation density map
- Concordance verification data for every BL match

At that point, multimodal RAG would add genuine retrieval power. Before
that point, it is premature infrastructure. The architecture described in
`docs/archive/HPMULTIMODAL.md` remains a valid reference for when the time
comes, but it is not a prerequisite for the work described in this plan.

## What This Won't Solve

Things that remain beyond automated image reading:

1. **Definitive hand attribution.** Distinguishing Hand A from Hand B
   requires ink color analysis that grayscale photos cannot support.

2. **Reliable alchemical symbol identification.** The symbols are 2-3mm in
   the original manuscript. At photograph resolution, they are a few pixels.
   A specialist with the physical book would produce better results.

3. **1545 vs 1499 collation verification.** Whether the 1545 reprint follows
   the 1499 gathering structure is a codicological question that image
   reading can support (by finding signature marks) but probably cannot
   resolve (most signatures are trimmed or illegible).

4. **Scholarly-quality transcription.** LLM transcription of 17th-century
   secretary hand produces useful first drafts, not publishable readings.
   All transcriptions require expert verification.
